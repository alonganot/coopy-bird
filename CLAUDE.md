# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser game built with TypeScript + Vite + React, with an optional real-time multiplayer mode backed by a small self-hosted Node/WebSocket+HTTP server and a Postgres database for all player save data. `npm run dev` for the client dev server (`--host`, so it's reachable over LAN), `npm run server` for the server (production-shaped: no `.env` auto-loading, matching Docker/Fly where env vars come from the container/host directly), `npm run server:dev` for local development (same thing but auto-loads `server/.env` via `tsx --env-file`), `npm run build` to type-check + produce a static `dist/`, `npm run typecheck:server` to type-check `server/` against `tsconfig.server.json`, `npm run test` for Vitest.

- `index.html` / `src/main.tsx` / `src/App.tsx` — thin Vite/React shell. `App.tsx` first renders `src/UsernameGate.tsx` (see Accounts & persistence below) until a session is established, then mounts either `createGame(canvas, controlsContainer, opts)` (single-player, `src/game/`) or `createMultiplayerGame(...)` (`src/multiplayer/`) in a `useEffect` gated on `ready`, keyed on a `mode` state toggle. React never re-renders the game itself after mount and owns no game state — all rendering is still hand-rolled Canvas2D, not DOM/JSX; the `controlsContainer` is where each engine imperatively mounts its own DOM overlays (mobile skill buttons, the mode-switch button, and — single-player only, see Settings below — the username-change field), synced once per frame the same way the canvas is.
- `src/game/` — the single-player game engine, plain `.ts` modules with no React dependency:
  - `constants.ts` — canvas size and physics constants (`GRAVITY`, `JUMP`, `PIPE_W`, `GAP`, `SPEED`, `DASH_SPEED_MULTIPLIER`, `W`, `H`). These are also imported directly by the multiplayer server (see below) since they're plain data with no DOM dependency.
  - `types.ts` — core types: `Bird`, `Pipe`, `Particle`, `Projectile`, `DecorParticle`, `Collidable`, `GameState`, `ShopTabId` (`'props' | 'skills' | 'pipes' | 'bg'`).
  - `state.ts` — the single mutable `world` object (bird/pipes/particles/projectiles/decor/score/state/gameData/username/shopState/shopTab/propsSubTab/settingsState/leaderboard/tick/glitch fields, etc.), plus `init()` and `makeDecor()`. This is the explicit, typed replacement for the old game.js/skills.js implicit global scope — nearly everything reads/writes through `world` directly, by design (no encapsulation layer, since every consumer needs direct field access). `world.gameData` starts as a safe local default (`migrateData({})`) and is replaced once `UsernameGate` resolves a real session; `world.username` is empty until then. `world.leaderboard` holds the last-fetched global single-player leaderboard (see Leaderboards below). `saveData()` now calls `api.ts`'s `pushGameData(world.username, world.gameData)` instead of writing to `localStorage` — none of its ~12 call sites (shop/skill equip & purchase, score/death) needed to change, since none of them ever awaited it.
  - `persistence.ts` — `GameData` interface (the save-data shape, now stored server-side — see Accounts & persistence below) plus `migrateData()` (defaults/backfills any missing/legacy field, applied to whatever the server returns for a username) and `getStoredUsername()`/`setStoredUsername()`, the only two things still touching `localStorage` — just the one username string this device last logged in as. `persistence.test.ts` (Vitest) covers `migrateData`/`migrateSkillData` edge cases — the one place a bug could silently lose a player's save.
  - `api.ts` — client-side fetch helpers for the server's REST API: `fetchOrCreateSession(username)` (called once by `UsernameGate`), `pushGameData(username, gameData)` (called by `state.ts`'s `saveData()`, fire-and-forget), `checkUsernameExists(username)`/`renameUsername(old, new)` (used by `UsernameGate`/the Settings screen — see Accounts & persistence below), and `submitScore`/`fetchLeaderboard(mode)` (see Leaderboards below). `resolveApiUrl()` mirrors `multiplayer/connection.ts`'s `resolveServerUrl()` pattern — defaults to the page's own hostname, overridable via `VITE_API_URL`.
  - `scoring.ts` — `onScore`/`onDeath`/`spawnPipeBreakParticles`. `onDeath()` also submits the run's score to the global single-player leaderboard and re-fetches it (fire-and-forget — see Leaderboards below).
  - `update.ts` — `update()` (physics/scoring/collision tick), `collides()`, `spawnPipe()`.
  - `input.ts` — `attachInput(canvas)` wires keyboard (`Space` jumps; `Digit1`/`Digit2`/`Digit3` activate the skill in that equip slot) and canvas click/touch (coordinates mapped from CSS pixels to the logical 400×600 space via `toCanvasCoords`, since the canvas is CSS-scaled — see Responsive/mobile below), returns a detach function. `jump()`, `handleCanvasClick()`, and the shared `pressSkillSlot`/`releaseSkillSlot` helpers (also reused by the mobile on-screen skill buttons) live here too.
  - `mobileControls.ts` — `attachSkillButtons(container, adapter)`: creates up to 3 on-screen touch buttons for skill slots, feature-detected (no-ops on non-touch devices). Generic over an `{ press, release, getSlot }` adapter so both `engine.ts` (single-player) and `multiplayer/mpEngine.ts` can reuse the same DOM/CSS without duplicating it — each engine supplies its own adapter built from its own state.
  - `modeSwitchButton.ts` — `attachModeSwitch(container, adapter)`: same adapter pattern, one DOM button whose position/size/visibility (`getPlacement()`) is driven per-frame by the calling engine, so it can be made to look "built into" whichever card is currently on screen (single-player's idle-screen shop button, or multiplayer's lobby READY button) without any canvas-drawing code needing to know about it.
  - `settingsPanel.ts`/`settingsInput.ts` — the Settings screen (idle state only — see Accounts & persistence below): a small "⚙" button (`drawSettingsButton`/`handleSettingsButtonClick`) fixed in the top-right corner, independent of the idle screen's title-card button stack; opening it (`world.settingsState`) draws a `neonPanel` overlay (`drawSettingsPanel`/`handleSettingsPanelClick`, Close button only) with a DOM username field on top (`settingsInput.ts`'s `attachSettingsInput`, adapted from the deleted `nameInput.ts` pattern but `async`-aware, since committing a rename is a server round-trip that can be rejected). `createSettingsInputAdapter()` wires the field to `api.ts`'s `renameUsername`, showing "USERNAME ALREADY TAKEN" inline (reusing the shop's fading-message convention) rather than closing the panel on conflict.
  - `engine.ts` — `createGame(canvas, controlsContainer?, options?)` owns the `ctx` (DPR-scaled backing store), the `requestAnimationFrame` loop, and `start()`/`stop()`. Each frame: draw background → pipes → bird → clone (if alive) → projectiles → particles → HUD → overlay → shop (if open) → scanlines → `update()` → sync mobile/mode-switch DOM overlays. `options.onSwitchMode` wires up the mode-switch button.
  - `render/` — draw functions grouped by visual concern, each taking `ctx: CanvasRenderingContext2D` as its first argument (no shared ctx global): `background.ts` (`drawBg`), `pipe.ts` (`drawPipe`, `drawPipePreview`), `bird.ts` (`drawBird` — takes an explicit `BirdRenderParams` object rather than reading `world`/`skillState` globals, specifically so `src/multiplayer/render/remoteBird.ts` can call the same function once per remote player; the single-player call site in `engine.ts` builds that params object from `world` itself, so this was a behavior-preserving extraction, not a visual change), `props.ts` (`drawHat`/`drawGlasses`/`drawMask`/`drawShoes` — see Props shop below), `particles.ts` (`drawParticles`, `drawProjectiles`, `drawCoin`), `hud.ts` (`drawHUD`), `overlay.ts` (`drawOverlay`, `drawScanlines`), `panel.ts` (`neonPanel`, shared by shop + overlay + multiplayer's lobby/match-ended screens).
  - `shop/` — `types.ts` (`ColorItem`, `PipeItem`, `BackgroundItem`, `ShopTabDef`, `GradientStop`, `CapStyle`, `DecorKind`, `PropsSubTabId`, `HatItem`/`GlassesItem`/`MaskItem`/`ShoeItem`), `data.ts` (`SHOP_COLORS`/`SHOP_PIPES`/`SHOP_BACKGROUNDS`/`SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES`/`SHOP_TABS`/`PROPS_SUB_TABS` plus pure `getActiveColorItem(gameData)`/`getActivePipeItem(gameData)`/`getActiveBackgroundItem(gameData)`/`getActiveHatItem(gameData)`/`getActiveGlassesItem(gameData)`/`getActiveMaskItem(gameData)`/`getActiveShoeItem(gameData)` helpers — these take `gameData` as a parameter rather than importing `world`, to avoid a circular import with `state.ts`), `collectibleShop.ts` (generic `purchaseOrEquip(item, unlockedList, activeKey)` shared by colors/pipes/backgrounds — always exactly one active item per category), `propGridTab.ts` (generic `drawPropGrid`/`handlePropGridClick`/`drawMiniBirdPreview` shared by the 4 wearable-prop sub-tabs, since their 2-column-grid layout is identical and only the preview/data differ — unlike `purchaseOrEquip`, clicking an already-equipped item here *unequips* it, since "nothing equipped" is a valid default per prop slot), `colorsTab.ts`/`pipesTab.ts`/`backgroundsTab.ts` (per-category draw + click, each its own file since the layouts genuinely differ), `hatsTab.ts`/`glassesTab.ts`/`masksTab.ts`/`shoesTab.ts` (thin wrappers around `propGridTab.ts` supplying each category's data + preview draw call), `propsTab.ts` (`drawPropsTab`/`handlePropsShopClick` — draws the Props tab's 5 inner sub-tab buttons (Colors/Hats/Glasses/Masks/Shoes) and dispatches to the matching sub-drawer, mirroring `shop.ts`'s own tab dispatch one level down), `shop.ts` (`drawShop`, `handleShopClick`, `drawShopTabs`, `drawCloseButton`, `renderShopMessage`, `drawShopButton`, and the `TAB_DRAWERS`/`TAB_HANDLERS` lookup tables dispatching to all 4 top-level tabs: Props/Skills/Pipes/Scene).
  - `skills/` — peer to `shop/`, not nested under it, because skills carry gameplay/physics responsibilities (collision radius, clone physics, per-frame updates) beyond "a shop tab": `data.ts` (`SHOP_SKILLS`, `SkillId`, `MAX_EQUIPPED_SKILLS`), `state.ts` (`skillState` — charges/activeTimer/hoverHeld/clone/shopMessage(+timer) — plus `initSkillState()`/`migrateSkillData()`), `skills.ts` (one cohesive module: `isSkillEquipped`/`isSkillActive`/`isSkillVisuallyActive`/`activateSkill`/`startHover`/`stopHover`/`getBirdRadius`/`mirrorCloneJump`/`attemptDeath`/`drawClone`/`updateSkills`, plus the internal per-skill effects for shooting/earthquake/shadowClone), `skillsTab.ts` (`drawSkillsTab`/`handleSkillsShopClick`). The server (`server/skills.ts`) reimplements this same logic standalone for multiplayer — see below.

Module coupling mirrors the original call graph directly (plain named-export imports at the same call sites, no event bus/DI/callback registry) — including a few intentional two-way import pairs where the original code was mutually recursive: `skills/skills.ts` ↔ `update.ts` (`collides`/`getBirdRadius`) and `skills/skills.ts` ↔ `scoring.ts` (`onDeath`, `spawnPipeBreakParticles` / `isSkillEquipped`). These work because all cross-references happen inside function bodies, never at module-top-level evaluation time.

## Responsive / mobile

The canvas is a fixed 400×600 *logical* coordinate space (all draw calls and hit-testing use `W`/`H` from `constants.ts`) that's CSS-scaled to fit any viewport — `src/style.css`'s `canvas` rule uses `aspect-ratio: 400/600` plus `width/height: min(100vw, calc(100dvh * 400/600))` (and its inverse) to letterbox correctly on any aspect ratio with pure CSS, no JS resize listener. `engine.ts`/`mpEngine.ts` additionally size the canvas's backing store to `W*devicePixelRatio`×`H*devicePixelRatio` and call `ctx.scale(dpr, dpr)` for crisp rendering on high-DPI phone screens; because of this, `input.ts`'s `toCanvasCoords()` (and `multiplayer/mpInput.ts`'s equivalent) deliberately maps click/touch coordinates against `W`/`H`, not `canvas.width`/`height`, since the latter includes the DPR multiplier. `index.html` has the standard mobile viewport meta tag (`user-scalable=no`, `viewport-fit=cover`), and `style.css` sets `touch-action: none`/`overscroll-behavior: none` on `body`/`canvas` to stop the browser from intercepting taps for zoom/scroll/pull-to-refresh during play. On-screen skill buttons (`mobileControls.ts`) only render on touch-capable devices (feature-detected once).

## PWA

The game is installable to a phone's home screen via `vite-plugin-pwa` (configured in `vite.config.ts`), generating a Workbox service worker + `manifest.webmanifest` at build time. Scope is deliberately "installable + fast shell loading," not full offline gameplay — since accounts/saves/leaderboards all require the server (see Accounts & persistence below), there's no meaningful single-player-only offline mode to build toward. `workbox.globPatterns` only precaches the built static shell (JS/CSS/HTML/icons); no `runtimeCaching` rule is registered for the API/WS origin, so login/save/leaderboard requests always hit the network fresh, never a stale cache. `registerType: 'autoUpdate'` means a new deploy takes over silently on the next load, with no manual "update available" prompt to build/maintain. The service worker only runs in production builds/preview/the deployed site — `devOptions.enabled` is left at its default (off), since a caching SW fighting Vite's dev-server hot-reload is a well-known footgun.

- `public/icon-source.svg` — the one hand-authored app icon (simple flat-color bird matching the in-game palette): **transparent background, subject only** — full-bleed, no baked-in shape/backdrop, per Apple's App Icon guidelines (the OS applies its own mask/shape, and — Icon Composer/native apps only — its own "Liquid Glass" material behind the artwork on iOS 26+; a plain web `apple-touch-icon` can't access that layered compositing, but staying background-free keeps the artwork consistent with it). `pwa-assets.config.js` (repo root) drives generation: the `transparent` variant (`pwa-{64,192,512}x512.png`, favicon) inherits that transparency, while `maskable`/`apple` (`maskable-icon-512x512.png`, `apple-touch-icon-180x180.png`) explicitly override `resizeOptions.background` to the app's dark navy (`#010818`) — both of those get hard-cropped to a shape by their OS, so they can't be transparent (that would show through as a hole, not a backdrop). Regenerate via `npx pwa-assets-generator` (reads `pwa-assets.config.js` automatically) after editing `icon-source.svg` — a one-time asset-prep step, not run on every build; the generated PNGs are committed like any other static asset.
- `index.html` has the handful of tags the plugin doesn't auto-inject: `apple-touch-icon`/favicon `<link>`s and `apple-mobile-web-app-*`/`theme-color` `<meta>` tags (for iOS home-screen behavior, which doesn't read the web manifest for these). The `<link rel="manifest">` and the service-worker registration script (`registerSW.js`) are auto-injected by the plugin at build time.
- `src/UsernameGate.tsx`'s `connectionErrorMessage()` distinguishes "genuinely offline" (`!navigator.onLine`, or a `fetch` `TypeError`) from an ordinary server error, so a mobile player with no signal sees "You're offline — connect to the internet to play" instead of an ambiguous "could not connect."

## Multiplayer

Real-time multiplayer is a fully separate, additive mode — nothing in `src/game/` branches on a mode flag (aside from the `drawBird` extraction noted above, behavior-preserving for single-player). It's **server-authoritative**: a small self-hosted Node + `ws` process (`server/`, run via `tsx`, no build step) owns one global room's entire physics/pipes/scoring/skills simulation and broadcasts full-state snapshots to every connected client at 30 Hz; clients are thin input-senders + renderers. This design was chosen specifically because shared-pipe-affecting skills (Dash/Earthquake/Shooting) mean multiple players' actions mutate one shared pipe array — a single authoritative process removes any possibility of cross-client desync, since only one process ever touches that array.

- `server/protocol.ts` — the JSON message shapes shared between client and server (`ClientMessage`/`ServerMessage` unions, `SnapshotPlayer`, `PlayerProps`, `LeaderboardEntry`), plus tuning constants (`TICK_RATE_HZ`, `READY_COUNTDOWN_MS`, `RESPAWN_DELAY_MS`, `INVULNERABILITY_MS`). Imported directly by `src/multiplayer/` too, since it's plain types/constants. `SnapshotPlayer.props`/the `join` message's `props` field carry only the 4 equipped prop ids (`hat`/`glasses`/`mask`/`shoe`), not styling data — same as `color` is sent as resolved values but props stay ids so the client resolves them via its own `SHOP_HATS`/etc. lookups, keeping the wire payload small. As with `color`/`equippedOrder`, props are only sent once at `join()` time — there's no live-update-while-connected message for any cosmetic yet, so changing props mid-session requires reconnecting.
- `server/room.ts` — `Room`/`ServerPlayer` in-memory state (one room, no DB) and small pure helpers (`aliveCount`, `allReady`, `findRespawnReference`, `resetRoomToWaiting`).
- `server/collision.ts` — a standalone reimplementation of the client's `update.ts` `collides()` (can't import that module directly since it's coupled to the browser-singleton `state.ts`/`skills.ts`) — kept in sync with it manually; it's a ~5-line pure function.
- `server/skills.ts` — server-side reimplementation of `skills/skills.ts`'s effect logic (activate/hover/earthquake/clone/etc.) against `ServerPlayer`/`room.pipes` instead of the client's `world`/`skillState` singletons.
- `server/tick.ts` — `stepRoom(room, now, callbacks)`: the room phase machine (`waiting` → `ready-check` → `countdown` → `playing` → `ended`) and the 30 Hz `simulateTick()` (gravity → shared pipe speed → spawn/move pipes → per-player skill updates → scoring → collision → floor/ceiling → respawn processing). Requires an explicit ready-up even with only one player connected, and always routes back through `ready-check` after a match ends (no auto-start shortcut at any player count). If `room.players.size` hits 0 for any reason (last player disconnects, switches back to single-player, etc.), the room immediately resets to `waiting` regardless of what phase it was in.
- `server/leaderboard.ts` — Postgres-backed, mode-aware (see Leaderboards below): `submitScore(username, mode, score)`/`topLeaderboard(mode, n)` against the shared `leaderboard_best` table.
- `server/index.ts` — process entry: creates one `http.Server` (`0.0.0.0`, not just localhost) shared by both `handleHttp` (see `server/http.ts` below) and the `WebSocketServer` (`new WebSocketServer({ server })`, upgrading on the same port) — a single port is required for a single Fly.io app/machine. Calls `initSchema()` once before accepting connections, logs both the localhost and LAN-reachable URLs on boot (mirroring Vite's `--host` banner), handles `join`/`ready`/`jump`/`skillActivate`/`skillHoverStart`/`skillHoverStop`/`leaderboardRequest` WS messages, and runs the tick/broadcast interval. `onMatchEnd` submits each player's `name` (always their real account username — see Leaderboards below) and the match's shared score to the multiplayer leaderboard.
- `src/multiplayer/` — the client side, structured as a parallel tree to `src/game/`:
  - `connection.ts` — WebSocket wrapper with message queueing (so a `join` sent before the socket opens isn't dropped) and simple reconnect-with-backoff. Resolves its server URL from the page's own `window.location.hostname` by default (not a hardcoded `localhost`), so it works correctly when a LAN client loads the page via the host's IP — `VITE_MP_SERVER_URL`/`VITE_MP_SERVER_PORT` (`.env`) can override this.
  - `mpState.ts` — local mutable singleton holding the latest server snapshot + own player id (mirrors `game/state.ts`'s pattern, fully separate from it). `PLAYER_X = 80` here matches the fixed x every bird sits at server-side (birds don't move horizontally; only pipes scroll).
  - `mpInput.ts` — sends `{jump}`/`{skillActivate}`/`{skillHoverStart/Stop}`/`{ready}` messages instead of mutating local state; reuses the same `toCanvasCoords`-style logical-space coordinate mapping as single-player's `input.ts`.
  - `mpEngine.ts` — `createMultiplayerGame(canvas, controlsContainer?, options?)`: parallel to `game/engine.ts`. Joins immediately on `start()`, sending `world.username` directly as the `join` message's `name` — there's no separate lobby nickname or name prompt anymore (see Accounts & persistence below) — and attaches the mobile skill buttons and mode-switch button via the same adapter patterns `game/engine.ts` uses.
  - `render/remoteBird.ts` — calls the shared `drawBird()` once per snapshot player (see the `render/bird.ts` note above), resolving each `SnapshotPlayer.props` id into a full item via `SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES` before passing it into `drawBird`'s `props` param, plus a nametag and an invulnerability flash ring; Shadow Clone's second entity is drawn as a simplified marker.
  - `render/lobby.ts` — the ready-check screen. `computeLayout()` is the single source of truth for the panel's height and every row's position (title/message/player-list/READY button), so `getReadyButtonRect()` (used for both drawing and click hit-testing) and `mpEngine.ts`'s mode-switch placement all stay consistent as the player count changes. The roster (`p.name`, always the real account username) is read-only here — no in-lobby rename control.
  - `render/mpHud.ts` — shared score (center-top), a compact alive/dead roster (top-right), and the local player's own equipped-skill charge rows (top-left) — this last part mirrors single-player's `render/hud.ts` exactly, reading from the local player's own `SnapshotPlayer` fields instead of `skillState`.
  - `render/mpOverlay.ts` — match-ended screen: final shared score + top-10 leaderboard.

## Accounts & persistence

All player save data (`GameData` — coins, unlocks, high scores, equipped cosmetics) lives server-side in Postgres, keyed by a **unique username with no password** — deliberately minimal per product decision: a username is "claimed" simply by being the first to use it. `localStorage` now only remembers which username this device last used (`persistence.ts`'s `getStoredUsername`/`setStoredUsername`), not the save data itself. Username is the *one* identity everywhere — there's no separate multiplayer nickname (that concept, `GameData.displayName`, was removed; see Multiplayer above).

- `server/db.ts` — a `pg.Pool` from `DATABASE_URL`, plus `initSchema()` (idempotent `CREATE TABLE IF NOT EXISTS` for `players`/`leaderboard_best`, run once at boot — no separate migration tool at this scale).
- `server/players.ts` — `getOrCreatePlayer(username)` (row lookup, inserts a default `migrateData({})` row if new), `savePlayerData(username, gameData)` (upserts the `game_data` JSONB column — the entire `GameData` shape is stored as one blob rather than per-field columns, so `migrateData()` keeps working unchanged), `usernameExists(username)`, and `renamePlayer(old, new)` (a single `UPDATE players SET username = $new WHERE username = $old` — `leaderboard_best.username`'s `ON UPDATE CASCADE` foreign key, see Leaderboards below, propagates the new name into that table automatically; a duplicate `$new` throws a Postgres unique-violation, which callers translate into a "taken" response).
- `server/http.ts` — the REST half of the merged HTTP+WS server (see `server/index.ts` above): `GET /health` (Fly's health check), `POST /api/session` (`{username}` → validates via regex, `getOrCreatePlayer`, returns `{gameData}` — covers both first-time creation and every later page load/reload), `PUT /api/players/:username` (`{gameData}`, size-capped at 64KB, upserts), `GET /api/players/:username/exists` (read-only uniqueness check), `PUT /api/players/:username/rename` (`{newUsername}` → `409 {error:'taken'}` on conflict). CORS is handled here via `ALLOWED_ORIGIN` (falls back to `*`); no cookies/auth tokens are involved, so preflight handling stays simple.
- `src/UsernameGate.tsx` — shown by `App.tsx` before the canvas mounts. Reads a stored username and silently re-establishes that session (no re-confirmation — returning to a remembered username on this device is already an implicit "log in as this user"). For a freshly-typed username, it first calls `api.ts`'s `checkUsernameExists`: if free, creates+logs in as usual; if taken, shows a `'confirmTaken'` view ("already taken — log in as this user, or try a different name?") rather than silently taking over the existing account. Either way, a successful login calls `fetchOrCreateSession`, populates `world.gameData`/`world.username`, then signals `App.tsx` to mount the game.
- Changing your username (the only way to do so) is the Settings screen — see `settingsPanel.ts`/`settingsInput.ts` above — reachable only from the single-player idle screen, not from the multiplayer lobby.

## Leaderboards

Both modes have their own **global, cross-player, one-row-per-player** leaderboard — each player's *best* score only, not a history of every run.

- One Postgres table, `leaderboard_best (username, mode, score, achieved_at)`, primary key `(username, mode)`, `mode` constrained to `'singleplayer' | 'multiplayer'` — see `server/db.ts`. `username` is a foreign key to `players(username)` with `ON UPDATE CASCADE ON DELETE CASCADE`, which is what makes the Settings-screen rename above safe: renaming only ever touches the `players` row directly, and Postgres itself keeps this table in sync.
- `server/leaderboard.ts`'s `submitScore(username, mode, score)` is an upsert-if-higher (`ON CONFLICT ... DO UPDATE ... WHERE leaderboard_best.score < excluded.score`) — a no-op if the new score isn't a personal best, so no separate "is this better?" read-then-write is needed.
- **Single-player**: `scoring.ts`'s `onDeath()` fire-and-forgets `api.ts`'s `submitScore`/`fetchLeaderboard('singleplayer')`, populating `world.leaderboard`; `render/overlay.ts`'s death screen renders it (replacing what used to be a personal top-5 list — the idle screen's personal "BEST: {x}" and `GameData.highScores` itself are unrelated and untouched).
- **Multiplayer**: `server/index.ts`'s `onMatchEnd` calls `submitScore(p.name, 'multiplayer', finishedRoom.score)` per connected player. This relies on `p.name` always being the real account username now (never a free-text nickname) — see Multiplayer above — so it's already the correct, stable per-player key; no separate identity field was needed on `ServerPlayer`/in the `join` message. `mpOverlay.ts`'s match-ended screen and the `leaderboardRequest` WS message both read from the same table via `topLeaderboard('multiplayer', n)`.

## Deployment

- **Client**: static Vite build (`dist/`), deployed to **Cloudflare Pages** — build command `npm run build`, output directory `dist`, env vars `VITE_API_URL`/`VITE_MP_SERVER_URL` set to the deployed server's `https://`/`wss://` URL. No Docker involved; Pages builds directly from git.
- **Server**: `server/Dockerfile` (single-stage `node:20-alpine`, `npm ci --omit=dev`, runs via `npm run server` i.e. `tsx` — no compile step), deployed to **Fly.io** (`fly.toml`) as one always-on machine (`auto_stop_machines = false`, `min_machines_running = 1` — Fly's default template auto-stops idle machines, which would reintroduce cold-start delays otherwise). `DATABASE_URL`/`ALLOWED_ORIGIN` are set as Fly secrets, never baked into the image.
- **Database**: any Postgres works (`server/db.ts` just needs `DATABASE_URL`); a free-tier host like Neon decouples the data from the app's own compute/redeploys.
- **Local integration testing**: `docker-compose.yml` runs `postgres` + the `server` image together (not the client — run that with plain `npm run dev` against `.env.local` pointing at `http://localhost:8080`).
- `.env`/`server/.env` are gitignored; `.env.example`/`server/.env.example` document every variable.
- `.github/workflows/ci.yml` runs `npm run build && npm run typecheck:server && npm run test` on push/PR. Deploys themselves aren't automated here — Cloudflare Pages auto-deploys from git on its own, and Fly deploys are a manual `fly deploy`.

## Architecture (single-player)

- **Persistence** — see `persistence.ts`/`api.ts`/Accounts & persistence above. `GameData` fields: `totalCoins`, `highScores`, `unlockedColors`/`activeColor`, `unlockedPipes`/`activePipe`, `unlockedBackgrounds`/`activeBackground`, `unlockedHats`/`activeHat`, `unlockedGlasses`/`activeGlasses`, `unlockedMasks`/`activeMask`, `unlockedShoes`/`activeShoe` (each `activeX` defaults to `''` — unlike colors/pipes/backgrounds, "nothing equipped" is a valid default per prop slot), `unlockedSkills`, `activeSkills` (`Record<SkillId, boolean>`, always fully populated by `migrateSkillData` before use), `equippedOrder`. The whole object now lives server-side in Postgres, keyed by username, not in `localStorage`.
- **Game loop** — see `engine.ts` above.
- **State machine** — `world.state` is `'idle' | 'play' | 'dead'`. `world.shopState` is `null | 'shop'` and `world.settingsState` is `null | 'settings'`, both overlaying on top of `state === 'idle'` (mutually exclusive — see `settingsPanel.ts`/`shop.ts` above). `world.shopTab` is `'props' | 'skills' | 'pipes' | 'bg'` (tab defs in `SHOP_TABS`); `world.propsSubTab` is `'colors' | 'hats' | 'glasses' | 'masks' | 'shoes'` (tab defs in `PROPS_SUB_TABS`), only meaningful while `shopTab === 'props'`. This is entirely independent of multiplayer's server-driven room `phase` (`waiting`/`ready-check`/`countdown`/`playing`/`ended`, see Multiplayer above) — the two state machines never interact.
- **Shop** — see `shop/shop.ts` above; all 4 top-level tabs go through the same `TAB_DRAWERS`/`TAB_HANDLERS` dispatch, no special-cased fallback tab. The Props tab (`shop/propsTab.ts`) is itself a second-level dispatch over 5 sub-tabs (Colors + the 4 prop categories) using the same `TAB_DRAWERS`/`TAB_HANDLERS`-style `Record` pattern, one nesting level down.
- **Color shop** — `SHOP_COLORS` (`shop/data.ts`) defines all purchasable bird color schemes. `getActiveColorItem(gameData)` is called inside `render/bird.ts`'s `drawBird()`. Rendered as the `colors` sub-tab of the Props tab (`shop/colorsTab.ts`, reused unmodified).
- **Pipes shop** — `SHOP_PIPES` (`shop/data.ts`) defines purchasable pipe skins: `barrelColors`/`capColors` (gradient stop lists), `stripeColor`/`scanlineColor`/`barrelGlow`/`capGlow`/`edgeGlow` (accent colors), and `capStyle` (`'straight' | 'hazard-stripe' | 'rounded' | 'diamond-notch' | 'ringed-segments' | 'crystal-spike'`, switched on inside `render/pipe.ts`'s `drawCap()`). `getActivePipeItem(gameData)` is called inside `drawPipe()`.
- **Backgrounds shop** — `SHOP_BACKGROUNDS` (`shop/data.ts`) defines purchasable scene themes: `sky` (gradient stops), `floor` (`{top, bottom}`), `horizon`/`grid` colors, `decor` (`{kind, color, count}` — kind is `'stars' | 'fireflies' | 'bubbles' | 'sandhaze' | 'snowflakes' | 'embers'`, switched on inside `render/background.ts`'s `drawBg()`), and optional `glowAccent` (a static radial glow, e.g. desert's sun/volcano's lava). The `world.decor` particle array (regenerated by `makeDecor()` in `state.ts`, sized per-theme) drives all background ambience — same particle shape `{x,y,r,phase,speed}` for every theme, only the draw/motion branch differs. Equipping a theme calls `makeDecor()` to resize the particle array.
- **Props shop** — `SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES` (`shop/data.ts`) each define 8 purchasable wearables priced 100–2000, rendered as the remaining 4 sub-tabs of the Props tab (`shop/hatsTab.ts`/`glassesTab.ts`/`masksTab.ts`/`shoesTab.ts`, all thin wrappers over the shared `shop/propGridTab.ts` grid). Unlike colors/pipes/backgrounds, all 4 prop slots (hat, glasses, mask, shoes) can be equipped simultaneously since they don't visually overlap, and clicking an already-equipped prop unequips it (sets the matching `activeX` back to `''`) rather than always leaving one item active — `propGridTab.ts`'s `handlePropGridClick` implements this toggle, deliberately not reusing `collectibleShop.ts`'s `purchaseOrEquip`. `render/props.ts`'s `drawHat`/`drawGlasses`/`drawMask`/`drawShoes` are called from inside `render/bird.ts`'s `drawBird()`, in its already-translated/rotated/(shrink-)scaled local space, so props automatically inherit the bird's pitch rotation and the Shrink skill's scale-down with no extra math; each function also applies its own `r / 18` scale so props stay correctly sized if a future caller ever passes a bird radius other than the constant 18. Insertion order inside `drawBird` (outermost to innermost): shoes → mask → (wing) → (eyes) → glasses → (beak) → hat, so shoes sit at the body's base, mask/glasses layer onto the face in front of the body but the glasses still sit on top of the drawn eyes, and the hat renders last, on top of everything, above the head.
- **Skills** (`skills/`) — `SHOP_SKILLS` defines 9 purchasable skills: Dash, Shooting, Invisibility, Pocket Dimension (time slow), Shrink, Hover, Earthquake, Freeze Frame, Shadow Clone. Max 3 equipped at once via `world.gameData.equippedOrder` (array of skill ids, slot number = index). Equipped skills gain a charge every `chargeThreshold` score points; `activateSkill(id)` consumes a charge and starts `skillState.activeTimer[id]`. Hover is hold-to-activate via `startHover()`/`stopHover()` instead of a single tap. `updateSkills()` runs once per frame from `update.ts`'s `update()` (after gravity/pipe movement) and handles timers, projectiles, earthquake pipe-sinking, and Shadow Clone physics. `attemptDeath()` replaces direct `onDeath()` calls in collision code — if a Shadow Clone is alive it swaps control to the clone instead of ending the run.
- **Input** — see `input.ts` above.
