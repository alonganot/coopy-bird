# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser game built with TypeScript + Vite + React. `npm run dev` for a dev server, `npm run build` to type-check and produce a static `dist/`, `npm run test` for Vitest.

- `index.html` / `src/main.tsx` / `src/App.tsx` — thin Vite/React shell. `App.tsx` renders a single `<canvas>` and hands it to `createGame(canvas)` in a `useEffect`; React never re-renders after mount and owns no game state — all rendering is still hand-rolled Canvas2D, not DOM/JSX.
- `src/game/` — the actual game engine, plain `.ts` modules with no React dependency:
  - `constants.ts` — canvas size and physics constants (`GRAVITY`, `JUMP`, `PIPE_W`, `GAP`, `SPEED`, `DASH_SPEED_MULTIPLIER`, `W`, `H`).
  - `types.ts` — core types: `Bird`, `Pipe`, `Particle`, `Projectile`, `DecorParticle`, `Collidable`, `GameState`, `ShopTabId`.
  - `state.ts` — the single mutable `world` object (bird/pipes/particles/projectiles/decor/score/state/gameData/shopState/shopTab/tick/glitch fields, etc.), plus `init()` and `makeDecor()`. This is the explicit, typed replacement for the old game.js/skills.js implicit global scope — nearly everything reads/writes through `world` directly, by design (no encapsulation layer, since every consumer needs direct field access).
  - `persistence.ts` — `GameData` interface (the localStorage save shape) and `loadData`/`saveData`/`migrateData`. `persistence.test.ts` (Vitest) covers `migrateData`/`migrateSkillData` edge cases — the one place a bug could silently lose a player's save.
  - `scoring.ts` — `onScore`/`onDeath`/`spawnPipeBreakParticles`.
  - `update.ts` — `update()` (physics/scoring/collision tick), `collides()`, `spawnPipe()`.
  - `input.ts` — `attachInput(canvas)` wires keyboard (`Space` jumps; `Digit1`/`Digit2`/`Digit3` activate the skill in that equip slot) and canvas click/touch, returns a detach function. `jump()` and `handleCanvasClick()` live here too.
  - `engine.ts` — `createGame(canvas)` owns the `ctx`, the `requestAnimationFrame` loop, and `start()`/`stop()`. Each frame: draw background → pipes → bird → clone (if alive) → projectiles → particles → HUD → overlay → shop (if open) → `update()`.
  - `render/` — draw functions grouped by visual concern, each taking `ctx: CanvasRenderingContext2D` as its first argument (no shared ctx global): `background.ts` (`drawBg`), `pipe.ts` (`drawPipe`, `drawPipePreview`), `bird.ts` (`drawBird`), `particles.ts` (`drawParticles`, `drawProjectiles`, `drawCoin`), `hud.ts` (`drawHUD`), `overlay.ts` (`drawOverlay`, `drawScanlines`), `panel.ts` (`neonPanel`, shared by shop + overlay).
  - `shop/` — `types.ts` (`ColorItem`, `PipeItem`, `BackgroundItem`, `ShopTabDef`, `GradientStop`, `CapStyle`, `DecorKind`), `data.ts` (`SHOP_COLORS`/`SHOP_PIPES`/`SHOP_BACKGROUNDS`/`SHOP_TABS` plus pure `getActiveColorItem(gameData)`/`getActivePipeItem(gameData)`/`getActiveBackgroundItem(gameData)` helpers — these take `gameData` as a parameter rather than importing `world`, to avoid a circular import with `state.ts`), `collectibleShop.ts` (generic `purchaseOrEquip(item, unlockedList, activeKey)` shared by colors/pipes/backgrounds), `colorsTab.ts`/`pipesTab.ts`/`backgroundsTab.ts` (per-category draw + click, each its own file since the layouts genuinely differ), `shop.ts` (`drawShop`, `handleShopClick`, `drawShopTabs`, `drawCloseButton`, `renderShopMessage`, `drawShopButton`, and the `TAB_DRAWERS`/`TAB_HANDLERS` lookup tables dispatching to all 4 tabs including skills).
  - `skills/` — peer to `shop/`, not nested under it, because skills carry gameplay/physics responsibilities (collision radius, clone physics, per-frame updates) beyond "a shop tab": `data.ts` (`SHOP_SKILLS`, `SkillId`, `MAX_EQUIPPED_SKILLS`), `state.ts` (`skillState` — charges/activeTimer/hoverHeld/clone/shopMessage(+timer) — plus `initSkillState()`/`migrateSkillData()`), `skills.ts` (one cohesive module: `isSkillEquipped`/`isSkillActive`/`isSkillVisuallyActive`/`activateSkill`/`startHover`/`stopHover`/`getBirdRadius`/`mirrorCloneJump`/`attemptDeath`/`drawClone`/`updateSkills`, plus the internal per-skill effects for shooting/earthquake/shadowClone), `skillsTab.ts` (`drawSkillsTab`/`handleSkillsShopClick`).

Module coupling mirrors the original call graph directly (plain named-export imports at the same call sites, no event bus/DI/callback registry) — including a few intentional two-way import pairs where the original code was mutually recursive: `skills/skills.ts` ↔ `update.ts` (`collides`/`getBirdRadius`) and `skills/skills.ts` ↔ `scoring.ts` (`onDeath`, `spawnPipeBreakParticles` / `isSkillEquipped`). These work because all cross-references happen inside function bodies, never at module-top-level evaluation time.

## Architecture

- **Persistence** — see `persistence.ts` above. `GameData` fields: `totalCoins`, `highScores`, `unlockedColors`/`activeColor`, `unlockedPipes`/`activePipe`, `unlockedBackgrounds`/`activeBackground`, `unlockedSkills`, `activeSkills` (`Record<SkillId, boolean>`, always fully populated by `migrateSkillData` before use), `equippedOrder`.
- **Game loop** — see `engine.ts` above.
- **State machine** — `world.state` is `'idle' | 'play' | 'dead'`. `world.shopState` is `null | 'shop'` and overlays on top of `state === 'idle'`. `world.shopTab` is `'colors' | 'skills' | 'pipes' | 'bg'` (tab defs in `SHOP_TABS`).
- **Shop** — see `shop/shop.ts` above; all 4 tabs (including Colors) go through the same `TAB_DRAWERS`/`TAB_HANDLERS` dispatch, no special-cased fallback tab.
- **Color shop** — `SHOP_COLORS` (`shop/data.ts`) defines all purchasable bird color schemes. `getActiveColorItem(gameData)` is called inside `render/bird.ts`'s `drawBird()`.
- **Pipes shop** — `SHOP_PIPES` (`shop/data.ts`) defines purchasable pipe skins: `barrelColors`/`capColors` (gradient stop lists), `stripeColor`/`scanlineColor`/`barrelGlow`/`capGlow`/`edgeGlow` (accent colors), and `capStyle` (`'straight' | 'hazard-stripe' | 'rounded' | 'diamond-notch' | 'ringed-segments' | 'crystal-spike'`, switched on inside `render/pipe.ts`'s `drawCap()`). `getActivePipeItem(gameData)` is called inside `drawPipe()`.
- **Backgrounds shop** — `SHOP_BACKGROUNDS` (`shop/data.ts`) defines purchasable scene themes: `sky` (gradient stops), `floor` (`{top, bottom}`), `horizon`/`grid` colors, `decor` (`{kind, color, count}` — kind is `'stars' | 'fireflies' | 'bubbles' | 'sandhaze' | 'snowflakes' | 'embers'`, switched on inside `render/background.ts`'s `drawBg()`), and optional `glowAccent` (a static radial glow, e.g. desert's sun/volcano's lava). The `world.decor` particle array (regenerated by `makeDecor()` in `state.ts`, sized per-theme) drives all background ambience — same particle shape `{x,y,r,phase,speed}` for every theme, only the draw/motion branch differs. Equipping a theme calls `makeDecor()` to resize the particle array.
- **Skills** (`skills/`) — `SHOP_SKILLS` defines 9 purchasable skills: Dash, Shooting, Invisibility, Pocket Dimension (time slow), Shrink, Hover, Earthquake, Freeze Frame, Shadow Clone. Max 3 equipped at once via `world.gameData.equippedOrder` (array of skill ids, slot number = index). Equipped skills gain a charge every `chargeThreshold` score points; `activateSkill(id)` consumes a charge and starts `skillState.activeTimer[id]`. Hover is hold-to-activate via `startHover()`/`stopHover()` instead of a single tap. `updateSkills()` runs once per frame from `update.ts`'s `update()` (after gravity/pipe movement) and handles timers, projectiles, earthquake pipe-sinking, and Shadow Clone physics. `attemptDeath()` replaces direct `onDeath()` calls in collision code — if a Shadow Clone is alive it swaps control to the clone instead of ending the run.
- **Input** — see `input.ts` above.
