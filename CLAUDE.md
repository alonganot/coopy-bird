# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A browser game built with TypeScript + Vite + React, with an optional real-time multiplayer mode backed by a small self-hosted Node/WebSocket server. `npm run dev` for the client dev server (`--host`, so it's reachable over LAN), `npm run server` for the multiplayer server, `npm run build` to type-check + produce a static `dist/`, `npm run test` for Vitest.

- `index.html` / `src/main.tsx` / `src/App.tsx` — thin Vite/React shell. `App.tsx` renders a single `<canvas>` plus a `controlsContainer` div, and mounts either `createGame(canvas, controlsContainer, opts)` (single-player, `src/game/`) or `createMultiplayerGame(...)` (`src/multiplayer/`) in a `useEffect`, keyed on a `mode` state toggle. React never re-renders after mount and owns no game state — all rendering is still hand-rolled Canvas2D, not DOM/JSX; the `controlsContainer` is where each engine imperatively mounts its own DOM overlays (mobile skill buttons, the mode-switch button, and — multiplayer only — the lobby name input), synced once per frame the same way the canvas is.
- `src/game/` — the single-player game engine, plain `.ts` modules with no React dependency:
  - `constants.ts` — canvas size and physics constants (`GRAVITY`, `JUMP`, `PIPE_W`, `GAP`, `SPEED`, `DASH_SPEED_MULTIPLIER`, `W`, `H`). These are also imported directly by the multiplayer server (see below) since they're plain data with no DOM dependency.
  - `types.ts` — core types: `Bird`, `Pipe`, `Particle`, `Projectile`, `DecorParticle`, `Collidable`, `GameState`, `ShopTabId` (`'props' | 'skills' | 'pipes' | 'bg'`).
  - `state.ts` — the single mutable `world` object (bird/pipes/particles/projectiles/decor/score/state/gameData/shopState/shopTab/propsSubTab/tick/glitch fields, etc.), plus `init()` and `makeDecor()`. This is the explicit, typed replacement for the old game.js/skills.js implicit global scope — nearly everything reads/writes through `world` directly, by design (no encapsulation layer, since every consumer needs direct field access).
  - `persistence.ts` — `GameData` interface (the localStorage save shape) and `loadData`/`saveData`/`migrateData`. `persistence.test.ts` (Vitest) covers `migrateData`/`migrateSkillData` edge cases — the one place a bug could silently lose a player's save. `GameData.displayName` (empty string = "not yet chosen") is the one field multiplayer added here — used for lobby nametags and the leaderboard.
  - `scoring.ts` — `onScore`/`onDeath`/`spawnPipeBreakParticles`.
  - `update.ts` — `update()` (physics/scoring/collision tick), `collides()`, `spawnPipe()`.
  - `input.ts` — `attachInput(canvas)` wires keyboard (`Space` jumps; `Digit1`/`Digit2`/`Digit3` activate the skill in that equip slot) and canvas click/touch (coordinates mapped from CSS pixels to the logical 400×600 space via `toCanvasCoords`, since the canvas is CSS-scaled — see Responsive/mobile below), returns a detach function. `jump()`, `handleCanvasClick()`, and the shared `pressSkillSlot`/`releaseSkillSlot` helpers (also reused by the mobile on-screen skill buttons) live here too.
  - `mobileControls.ts` — `attachSkillButtons(container, adapter)`: creates up to 3 on-screen touch buttons for skill slots, feature-detected (no-ops on non-touch devices). Generic over an `{ press, release, getSlot }` adapter so both `engine.ts` (single-player) and `multiplayer/mpEngine.ts` can reuse the same DOM/CSS without duplicating it — each engine supplies its own adapter built from its own state.
  - `modeSwitchButton.ts` — `attachModeSwitch(container, adapter)`: same adapter pattern, one DOM button whose position/size/visibility (`getPlacement()`) is driven per-frame by the calling engine, so it can be made to look "built into" whichever card is currently on screen (single-player's idle-screen shop button, or multiplayer's lobby READY button) without any canvas-drawing code needing to know about it.
  - `engine.ts` — `createGame(canvas, controlsContainer?, options?)` owns the `ctx` (DPR-scaled backing store), the `requestAnimationFrame` loop, and `start()`/`stop()`. Each frame: draw background → pipes → bird → clone (if alive) → projectiles → particles → HUD → overlay → shop (if open) → scanlines → `update()` → sync mobile/mode-switch DOM overlays. `options.onSwitchMode` wires up the mode-switch button.
  - `render/` — draw functions grouped by visual concern, each taking `ctx: CanvasRenderingContext2D` as its first argument (no shared ctx global): `background.ts` (`drawBg`), `pipe.ts` (`drawPipe`, `drawPipePreview`), `bird.ts` (`drawBird` — takes an explicit `BirdRenderParams` object rather than reading `world`/`skillState` globals, specifically so `src/multiplayer/render/remoteBird.ts` can call the same function once per remote player; the single-player call site in `engine.ts` builds that params object from `world` itself, so this was a behavior-preserving extraction, not a visual change), `props.ts` (`drawHat`/`drawGlasses`/`drawMask`/`drawShoes` — see Props shop below), `particles.ts` (`drawParticles`, `drawProjectiles`, `drawCoin`), `hud.ts` (`drawHUD`), `overlay.ts` (`drawOverlay`, `drawScanlines`), `panel.ts` (`neonPanel`, shared by shop + overlay + multiplayer's lobby/match-ended screens).
  - `shop/` — `types.ts` (`ColorItem`, `PipeItem`, `BackgroundItem`, `ShopTabDef`, `GradientStop`, `CapStyle`, `DecorKind`, `PropsSubTabId`, `HatItem`/`GlassesItem`/`MaskItem`/`ShoeItem`), `data.ts` (`SHOP_COLORS`/`SHOP_PIPES`/`SHOP_BACKGROUNDS`/`SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES`/`SHOP_TABS`/`PROPS_SUB_TABS` plus pure `getActiveColorItem(gameData)`/`getActivePipeItem(gameData)`/`getActiveBackgroundItem(gameData)`/`getActiveHatItem(gameData)`/`getActiveGlassesItem(gameData)`/`getActiveMaskItem(gameData)`/`getActiveShoeItem(gameData)` helpers — these take `gameData` as a parameter rather than importing `world`, to avoid a circular import with `state.ts`), `collectibleShop.ts` (generic `purchaseOrEquip(item, unlockedList, activeKey)` shared by colors/pipes/backgrounds — always exactly one active item per category), `propGridTab.ts` (generic `drawPropGrid`/`handlePropGridClick`/`drawMiniBirdPreview` shared by the 4 wearable-prop sub-tabs, since their 2-column-grid layout is identical and only the preview/data differ — unlike `purchaseOrEquip`, clicking an already-equipped item here *unequips* it, since "nothing equipped" is a valid default per prop slot), `colorsTab.ts`/`pipesTab.ts`/`backgroundsTab.ts` (per-category draw + click, each its own file since the layouts genuinely differ), `hatsTab.ts`/`glassesTab.ts`/`masksTab.ts`/`shoesTab.ts` (thin wrappers around `propGridTab.ts` supplying each category's data + preview draw call), `propsTab.ts` (`drawPropsTab`/`handlePropsShopClick` — draws the Props tab's 5 inner sub-tab buttons (Colors/Hats/Glasses/Masks/Shoes) and dispatches to the matching sub-drawer, mirroring `shop.ts`'s own tab dispatch one level down), `shop.ts` (`drawShop`, `handleShopClick`, `drawShopTabs`, `drawCloseButton`, `renderShopMessage`, `drawShopButton`, and the `TAB_DRAWERS`/`TAB_HANDLERS` lookup tables dispatching to all 4 top-level tabs: Props/Skills/Pipes/Scene).
  - `skills/` — peer to `shop/`, not nested under it, because skills carry gameplay/physics responsibilities (collision radius, clone physics, per-frame updates) beyond "a shop tab": `data.ts` (`SHOP_SKILLS`, `SkillId`, `MAX_EQUIPPED_SKILLS`), `state.ts` (`skillState` — charges/activeTimer/hoverHeld/clone/shopMessage(+timer) — plus `initSkillState()`/`migrateSkillData()`), `skills.ts` (one cohesive module: `isSkillEquipped`/`isSkillActive`/`isSkillVisuallyActive`/`activateSkill`/`startHover`/`stopHover`/`getBirdRadius`/`mirrorCloneJump`/`attemptDeath`/`drawClone`/`updateSkills`, plus the internal per-skill effects for shooting/earthquake/shadowClone), `skillsTab.ts` (`drawSkillsTab`/`handleSkillsShopClick`). The server (`server/skills.ts`) reimplements this same logic standalone for multiplayer — see below.

Module coupling mirrors the original call graph directly (plain named-export imports at the same call sites, no event bus/DI/callback registry) — including a few intentional two-way import pairs where the original code was mutually recursive: `skills/skills.ts` ↔ `update.ts` (`collides`/`getBirdRadius`) and `skills/skills.ts` ↔ `scoring.ts` (`onDeath`, `spawnPipeBreakParticles` / `isSkillEquipped`). These work because all cross-references happen inside function bodies, never at module-top-level evaluation time.

## Responsive / mobile

The canvas is a fixed 400×600 *logical* coordinate space (all draw calls and hit-testing use `W`/`H` from `constants.ts`) that's CSS-scaled to fit any viewport — `src/style.css`'s `canvas` rule uses `aspect-ratio: 400/600` plus `width/height: min(100vw, calc(100dvh * 400/600))` (and its inverse) to letterbox correctly on any aspect ratio with pure CSS, no JS resize listener. `engine.ts`/`mpEngine.ts` additionally size the canvas's backing store to `W*devicePixelRatio`×`H*devicePixelRatio` and call `ctx.scale(dpr, dpr)` for crisp rendering on high-DPI phone screens; because of this, `input.ts`'s `toCanvasCoords()` (and `multiplayer/mpInput.ts`'s equivalent) deliberately maps click/touch coordinates against `W`/`H`, not `canvas.width`/`height`, since the latter includes the DPR multiplier. `index.html` has the standard mobile viewport meta tag (`user-scalable=no`, `viewport-fit=cover`), and `style.css` sets `touch-action: none`/`overscroll-behavior: none` on `body`/`canvas` to stop the browser from intercepting taps for zoom/scroll/pull-to-refresh during play. On-screen skill buttons (`mobileControls.ts`) only render on touch-capable devices (feature-detected once).

## Multiplayer

Real-time multiplayer is a fully separate, additive mode — nothing in `src/game/` branches on a mode flag (aside from the `drawBird` extraction and the `displayName` field noted above, both behavior-preserving for single-player). It's **server-authoritative**: a small self-hosted Node + `ws` process (`server/`, run via `tsx`, no build step) owns one global room's entire physics/pipes/scoring/skills simulation and broadcasts full-state snapshots to every connected client at 30 Hz; clients are thin input-senders + renderers. This design was chosen specifically because shared-pipe-affecting skills (Dash/Earthquake/Shooting) mean multiple players' actions mutate one shared pipe array — a single authoritative process removes any possibility of cross-client desync, since only one process ever touches that array.

- `server/protocol.ts` — the JSON message shapes shared between client and server (`ClientMessage`/`ServerMessage` unions, `SnapshotPlayer`, `PlayerProps`, `LeaderboardEntry`), plus tuning constants (`TICK_RATE_HZ`, `READY_COUNTDOWN_MS`, `RESPAWN_DELAY_MS`, `INVULNERABILITY_MS`). Imported directly by `src/multiplayer/` too, since it's plain types/constants. `SnapshotPlayer.props`/the `join` message's `props` field carry only the 4 equipped prop ids (`hat`/`glasses`/`mask`/`shoe`), not styling data — same as `color` is sent as resolved values but props stay ids so the client resolves them via its own `SHOP_HATS`/etc. lookups, keeping the wire payload small. As with `color`/`equippedOrder`, props are only sent once at `join()` time — there's no live-update-while-connected message for any cosmetic yet, so changing props mid-session requires reconnecting.
- `server/room.ts` — `Room`/`ServerPlayer` in-memory state (one room, no DB) and small pure helpers (`aliveCount`, `allReady`, `findRespawnReference`, `resetRoomToWaiting`).
- `server/collision.ts` — a standalone reimplementation of the client's `update.ts` `collides()` (can't import that module directly since it's coupled to the browser-singleton `state.ts`/`skills.ts`) — kept in sync with it manually; it's a ~5-line pure function.
- `server/skills.ts` — server-side reimplementation of `skills/skills.ts`'s effect logic (activate/hover/earthquake/clone/etc.) against `ServerPlayer`/`room.pipes` instead of the client's `world`/`skillState` singletons.
- `server/tick.ts` — `stepRoom(room, now, callbacks)`: the room phase machine (`waiting` → `ready-check` → `countdown` → `playing` → `ended`) and the 30 Hz `simulateTick()` (gravity → shared pipe speed → spawn/move pipes → per-player skill updates → scoring → collision → floor/ceiling → respawn processing). Requires an explicit ready-up even with only one player connected, and always routes back through `ready-check` after a match ends (no auto-start shortcut at any player count). If `room.players.size` hits 0 for any reason (last player disconnects, switches back to single-player, etc.), the room immediately resets to `waiting` regardless of what phase it was in.
- `server/leaderboard.ts` — flat JSON file (`server/data/leaderboard.json`, gitignored) of `{name, score, timestamp}`, capped to top 100; written once per match end, read on demand. No DB — there's no concurrent-writer problem at this scale (one process, one room).
- `server/index.ts` — process entry: binds the `WebSocketServer` to `0.0.0.0` (not just localhost) and logs both the localhost and LAN-reachable URLs on boot (mirroring Vite's `--host` banner), handles `join`/`rename`/`ready`/`jump`/`skillActivate`/`skillHoverStart`/`skillHoverStop`/`leaderboardRequest` messages, and runs the tick/broadcast interval.
- `src/multiplayer/` — the client side, structured as a parallel tree to `src/game/`:
  - `connection.ts` — WebSocket wrapper with message queueing (so a `join` sent before the socket opens isn't dropped) and simple reconnect-with-backoff. Resolves its server URL from the page's own `window.location.hostname` by default (not a hardcoded `localhost`), so it works correctly when a LAN client loads the page via the host's IP — `VITE_MP_SERVER_URL`/`VITE_MP_SERVER_PORT` (`.env`) can override this.
  - `mpState.ts` — local mutable singleton holding the latest server snapshot + own player id (mirrors `game/state.ts`'s pattern, fully separate from it). `PLAYER_X = 80` here matches the fixed x every bird sits at server-side (birds don't move horizontally; only pipes scroll).
  - `mpInput.ts` — sends `{jump}`/`{skillActivate}`/`{skillHoverStart/Stop}`/`{ready}` messages instead of mutating local state; reuses the same `toCanvasCoords`-style logical-space coordinate mapping as single-player's `input.ts`.
  - `mpEngine.ts` — `createMultiplayerGame(canvas, controlsContainer?, options?)`: parallel to `game/engine.ts`. Joins immediately on `start()` (auto-generating + persisting a `PlayerNNNN` display name if none is set yet, so joining never blocks on a name prompt), and attaches the mobile skill buttons, mode-switch button, and the persistent lobby name input via the same adapter patterns `game/engine.ts` uses.
  - `nameInput.ts` — a persistent (not one-shot) DOM text input shown only during the `waiting`/`ready-check` lobby phases; committing a new name (blur/Enter) updates `GameData.displayName`, persists it, and sends a `rename` message so it propagates live to every other connected client's snapshot.
  - `render/remoteBird.ts` — calls the shared `drawBird()` once per snapshot player (see the `render/bird.ts` note above), resolving each `SnapshotPlayer.props` id into a full item via `SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES` before passing it into `drawBird`'s `props` param, plus a nametag and an invulnerability flash ring; Shadow Clone's second entity is drawn as a simplified marker.
  - `render/lobby.ts` — the ready-check screen. `computeLayout()` is the single source of truth for the panel's height and every row's position (title/message/name-input/player-list/READY button), so `getReadyButtonRect()`/`getNameInputRect()` (used for both drawing and click hit-testing) and `mpEngine.ts`'s mode-switch placement all stay consistent as the player count changes.
  - `render/mpHud.ts` — shared score (center-top), a compact alive/dead roster (top-right), and the local player's own equipped-skill charge rows (top-left) — this last part mirrors single-player's `render/hud.ts` exactly, reading from the local player's own `SnapshotPlayer` fields instead of `skillState`.
  - `render/mpOverlay.ts` — match-ended screen: final shared score + top-10 leaderboard.

## Architecture (single-player)

- **Persistence** — see `persistence.ts` above. `GameData` fields: `totalCoins`, `highScores`, `unlockedColors`/`activeColor`, `unlockedPipes`/`activePipe`, `unlockedBackgrounds`/`activeBackground`, `unlockedHats`/`activeHat`, `unlockedGlasses`/`activeGlasses`, `unlockedMasks`/`activeMask`, `unlockedShoes`/`activeShoe` (each `activeX` defaults to `''` — unlike colors/pipes/backgrounds, "nothing equipped" is a valid default per prop slot), `unlockedSkills`, `activeSkills` (`Record<SkillId, boolean>`, always fully populated by `migrateSkillData` before use), `equippedOrder`, `displayName` (used by multiplayer only).
- **Game loop** — see `engine.ts` above.
- **State machine** — `world.state` is `'idle' | 'play' | 'dead'`. `world.shopState` is `null | 'shop'` and overlays on top of `state === 'idle'`. `world.shopTab` is `'props' | 'skills' | 'pipes' | 'bg'` (tab defs in `SHOP_TABS`); `world.propsSubTab` is `'colors' | 'hats' | 'glasses' | 'masks' | 'shoes'` (tab defs in `PROPS_SUB_TABS`), only meaningful while `shopTab === 'props'`. This is entirely independent of multiplayer's server-driven room `phase` (`waiting`/`ready-check`/`countdown`/`playing`/`ended`, see Multiplayer above) — the two state machines never interact.
- **Shop** — see `shop/shop.ts` above; all 4 top-level tabs go through the same `TAB_DRAWERS`/`TAB_HANDLERS` dispatch, no special-cased fallback tab. The Props tab (`shop/propsTab.ts`) is itself a second-level dispatch over 5 sub-tabs (Colors + the 4 prop categories) using the same `TAB_DRAWERS`/`TAB_HANDLERS`-style `Record` pattern, one nesting level down.
- **Color shop** — `SHOP_COLORS` (`shop/data.ts`) defines all purchasable bird color schemes. `getActiveColorItem(gameData)` is called inside `render/bird.ts`'s `drawBird()`. Rendered as the `colors` sub-tab of the Props tab (`shop/colorsTab.ts`, reused unmodified).
- **Pipes shop** — `SHOP_PIPES` (`shop/data.ts`) defines purchasable pipe skins: `barrelColors`/`capColors` (gradient stop lists), `stripeColor`/`scanlineColor`/`barrelGlow`/`capGlow`/`edgeGlow` (accent colors), and `capStyle` (`'straight' | 'hazard-stripe' | 'rounded' | 'diamond-notch' | 'ringed-segments' | 'crystal-spike'`, switched on inside `render/pipe.ts`'s `drawCap()`). `getActivePipeItem(gameData)` is called inside `drawPipe()`.
- **Backgrounds shop** — `SHOP_BACKGROUNDS` (`shop/data.ts`) defines purchasable scene themes: `sky` (gradient stops), `floor` (`{top, bottom}`), `horizon`/`grid` colors, `decor` (`{kind, color, count}` — kind is `'stars' | 'fireflies' | 'bubbles' | 'sandhaze' | 'snowflakes' | 'embers'`, switched on inside `render/background.ts`'s `drawBg()`), and optional `glowAccent` (a static radial glow, e.g. desert's sun/volcano's lava). The `world.decor` particle array (regenerated by `makeDecor()` in `state.ts`, sized per-theme) drives all background ambience — same particle shape `{x,y,r,phase,speed}` for every theme, only the draw/motion branch differs. Equipping a theme calls `makeDecor()` to resize the particle array.
- **Props shop** — `SHOP_HATS`/`SHOP_GLASSES`/`SHOP_MASKS`/`SHOP_SHOES` (`shop/data.ts`) each define 8 purchasable wearables priced 100–2000, rendered as the remaining 4 sub-tabs of the Props tab (`shop/hatsTab.ts`/`glassesTab.ts`/`masksTab.ts`/`shoesTab.ts`, all thin wrappers over the shared `shop/propGridTab.ts` grid). Unlike colors/pipes/backgrounds, all 4 prop slots (hat, glasses, mask, shoes) can be equipped simultaneously since they don't visually overlap, and clicking an already-equipped prop unequips it (sets the matching `activeX` back to `''`) rather than always leaving one item active — `propGridTab.ts`'s `handlePropGridClick` implements this toggle, deliberately not reusing `collectibleShop.ts`'s `purchaseOrEquip`. `render/props.ts`'s `drawHat`/`drawGlasses`/`drawMask`/`drawShoes` are called from inside `render/bird.ts`'s `drawBird()`, in its already-translated/rotated/(shrink-)scaled local space, so props automatically inherit the bird's pitch rotation and the Shrink skill's scale-down with no extra math; each function also applies its own `r / 18` scale so props stay correctly sized if a future caller ever passes a bird radius other than the constant 18. Insertion order inside `drawBird` (outermost to innermost): shoes → mask → (wing) → (eyes) → glasses → (beak) → hat, so shoes sit at the body's base, mask/glasses layer onto the face in front of the body but the glasses still sit on top of the drawn eyes, and the hat renders last, on top of everything, above the head.
- **Skills** (`skills/`) — `SHOP_SKILLS` defines 9 purchasable skills: Dash, Shooting, Invisibility, Pocket Dimension (time slow), Shrink, Hover, Earthquake, Freeze Frame, Shadow Clone. Max 3 equipped at once via `world.gameData.equippedOrder` (array of skill ids, slot number = index). Equipped skills gain a charge every `chargeThreshold` score points; `activateSkill(id)` consumes a charge and starts `skillState.activeTimer[id]`. Hover is hold-to-activate via `startHover()`/`stopHover()` instead of a single tap. `updateSkills()` runs once per frame from `update.ts`'s `update()` (after gravity/pipe movement) and handles timers, projectiles, earthquake pipe-sinking, and Shadow Clone physics. `attemptDeath()` replaces direct `onDeath()` calls in collision code — if a Shadow Clone is alive it swaps control to the clone instead of ending the run.
- **Input** — see `input.ts` above.
